Debug Frontend 404（微服务）

系统性排查微服务架构中前端接口 404：前端代理 rewrite → 网关路由 → Controller 路径三层链路追踪，自动判定是「网关层错配」还是「前端 proxy rewrite 错配」。

从开始到完成，每一步做什么、产出什么一目了然。下方时间轴为逐步详情。

执行全流程（5 步 SOP）

00 检查前端代理 rewrite（dev 必做）
动作
读 vite/webpack 的 proxy.rewrite，确认浏览器路径经 rewrite 后是否等于网关路由前缀。
关键
浏览器路径前缀 ≠ 网关路由前缀，必须用 rewrite 后路径匹配网关。
01 全项目搜索 Controller
动作
grep 定位 Controller 所在微服务、@RequestMapping 完整路径、方法注解。
02 检查网关路由配置
动作
查网关 application.yml / routes.json / Nginx 的 path pattern 与 strip-prefix，用 rewrite 后路径匹配路由。
03 链路对比表（必填）
动作
列：浏览器路径 / rewrite 后路径 / 网关匹配路由 / 实际路由服务 / Controller 所在服务 → 是否一致。
判定
一致→其他原因；不一致→根因（网关层 / proxy rewrite 两类错配）。
04 给出结论
产出
诊断报告 开发过程/<服务名>-404问题-<日期>.md：链路对比表 + Mermaid 双链路图 + 错配类型判定 + 修复 diff（首选改前端）。

关键信息

触发条件

  • 前端接口返回 404
  • 网关路由到错误微服务
  • Controller 路径与网关前缀不匹配
  • Vite/Webpack proxy rewrite 目标前缀配错
  • 微服务架构 + 前端独立部署

排除边界

  • 前端页面路由 404
  • 后端 500 → debug-backend-500
  • 性能超时 / 数据库问题
  • 非微服务架构（无网关层）

产出物

  • 开发过程/<服务名>-404问题-<日期>.md
  • 链路对比表 + Mermaid 双链路图 + 修复 diff

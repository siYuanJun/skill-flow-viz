③ 工程过程层

AI-DEV-OPRO

AI Dev Workflow（ai-dev-workflow v2.5.1）——人工确认闭环门控的工程化开发脚手架：把需求转成完整项目，强制逐模块「规格确认→开发→自审→测试→熔断→闭环」。v2.5 核心：模块规格文档 + 逐模块人工确认 + 入口级硬约束；任务原子化拆分 + 断点续接（N+1）防上下文溢出；同一任务重试超 3 次触发熔断。杜绝 AI 跳流程。

执行全流程（5 步 + 逐模块闸控循环）

01 初始化与配置确认🛑 人工确认
产出
workflow-state.json（需求 / 技术栈偏好 / 熔断配置）
门控
交互式询问并记录配置，🛑 人工确认后才进入下一步。
02 创建项目骨架🛑 人工确认
产出
标准目录（frontend/backend/.ai-workflow/）+ .gitignore + .env + docker-compose.yml
门控
骨架生成后 🛑 人工确认。
03 模块拆解与规格确认🛑 逐模块确认
产出
INDEX.md、MODULE-xxx-spec.md（每模块一份）、DOCS-INDEX.md 骨架、ARCHITECTURE-OVERVIEW.md 骨架、含 Mermaid 依赖图的任务列表
门控
v2.5 看家本领：展示全部模块规格，🛑 人工逐模块确认后才进入开发。
原子化 + 断点续接（N+1）：PRD 拆为约 100 份原子任务（单函数/单测试/单文档段），任一会话上下文将满时切新窗口，读 INDEX.md 定位首个 ⏳ 任务从 N+1 续，绝不从头重跑。
04 逐模块执行与闭环🛑 每模块闭环确认
产出
代码文件、测试报告、任务日志（logs/TASK-XXX.md）、模块闭环文档（flow/design/summary/checklist 四份）
门控
每模块：展示规格 → 🛑 确认 → 开发 → 自审 → 测试 → 熔断检查 → 闭环 → 日志；严禁跨模块。闭环后 🛑 暂停，人工确认通过才解锁下一模块。
熔断器：同一任务重试 ≤ 3 次，超过触发熔断、标记「受阻」，等待人工决策。日志不完整视为任务未完成（不得标「完成」）。
05 交付文档与经验沉淀🛑 人工确认
产出
三层交付文档（流程 / 设计说明 / 大白话摘要 / 检查清单 / ADR）、架构总览(增量)、经验知识库 LESSONS-LEARNED.md(增量)
门控
全套交付文档生成后 🛑 人工确认。

关键信息

触发条件

  • 开发完整管理系统：如「开发电商后台管理系统」
  • 构建 SaaS 应用：如「构建企业 SaaS 应用」
  • 初始化全栈项目：如「初始化 Vue + Laravel 全栈项目」
  • 按工程化流程开发 / 从 0 到 1 完整开发
  • 已有项目追加功能模块（自动跳过 Step 1/2）

排除边界

  • 仅写一段代码（非工程化流程）
  • 修复 Bug / 解释调试（非完整开发流程）
  • 设计或审查：「设计数据库表」「代码审查」
  • 部署运维 / 技术咨询（应由其他 Skill 接管）

产出物（写入 .ai-workflow/）

  • workflow-state.json（配置基线 + 熔断配置）
  • 项目骨架（目录 / .gitignore / .env / docker-compose.yml）
  • 模块规格文档（INDEX.md / MODULE-xxx-spec.md，🛑 逐模块确认）
  • 代码文件 + 测试报告 + 任务日志（logs/TASK-XXX.md）
  • 模块闭环文档（flow / design / summary / checklist）
  • 全套交付文档（流程 / 设计 / 大白话摘要 / 检查清单 / ADR）
  • 架构总览(增量) + 经验知识库 LESSONS-LEARNED.md(增量)

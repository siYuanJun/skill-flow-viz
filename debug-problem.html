Debug Problem（通用排查）

通用型技术问题排查入口：自动判定前端/后端 + 语言/框架，假设驱动逐步验证，适配 PHP/Java/Python/Go/Vue/React 任意技术栈。不限定 Docker 或特定框架。

从开始到完成，每一步做什么、产出什么一目了然。◆ 菱形为人工审批闸口（Gate）。下方时间轴为逐步详情。

执行全流程（4 步）

00 自动分类（无需用户指定）
判定
前端/后端（报错位置）、语言/框架（Laravel/Spring Boot/Django/Nuxt/Go）、运行环境（Docker/Web服务器/本地）、问题类型（HTTP码/堆栈/渲染/数据/性能）。
输出
一句话告知判断，如「后端 Java + Docker，HTTP 500」。
01 收集证据（按需自适应）
动作
按分类选证据方式（Docker: docker logs；本地: tail/grep；前端: Console/Network）。必收集：完整错误、触发条件、相关上下文。
02 假设驱动排查（核心）
动作
列 2-5 个原因（按可能性排序），逐个验证，更新结果（确认 / 排除 / 待深入）。每次只验一个假设。
03 定位根因 + 修复 + 验证
动作
根因一句话 + 代码 diff/配置调整（先确认再执行）+ 验证命令。最小改动优先。
产出
排查笔记存 .ai-workflow/debug-problem/[日期]-[问题简述].md；复杂问题可转 doc-troubleshooting 复盘。

关键信息

触发条件

  • 任何困惑的技术问题（报错/异常/行为不符预期）
  • 不确定前端还是后端问题
  • 技术栈不限（PHP/Java/Python/Go/Vue/React…）

排除边界

  • 需求分析/方案设计 → meta-feature-dev
  • 纯代码编写 → AI Dev Workflow
  • Docker+Java 专项 → debug-backend-500
  • 404 链路追踪 → debug-frontend-404
  • 前端参数排查文档 → debug-frontend-param

产出物

  • .ai-workflow/debug-problem/[日期]-[问题简述].md
  • 根因 + 修复 diff + 验证命令

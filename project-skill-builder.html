Project Skill Builder

项目 Skill 工程惯例书：五大惯例 + 四种设计模式 + 质检闭环，让新窗口 Agent 自动继承项目工程标准。创建新 Skill 时触发，全程指导目录布局、产出隔离、SKILL.md 规范与治理验收。

执行全流程（创建新 Skill 的 10 步）

01 确认名称与目录
动作
先查 核心Skill工程目录映射.md 确认无冲突，确定 Skill 名称，在 skills/ 下创建目录。
产出
skills/<skill-name>/ 空目录
02 写 SKILL.md 入口
动作
写 YAML frontmatter（name + description，含触发关键词）+ 标准结构块（这是什么 / 何时使用 / 流程 / 输入输出 / 关键原则 / 相关 Skill）。
红线
SKILL.md ≤ 1000 tokens；深度内容下沉到 references/。
03 选定设计模式
动作
四选一：蓝图型（docker-container-design）/ SOP型（debug-backend-500）/ 工具入口型（debug-problem）/ 元方法论型（meta-agent-collaboration），对照范例写流程。
04 规划产出隔离子树
动作
为非 ai-dev-workflow 的 Skill 申请 .ai-workflow/<skill-name>/ 独立子目录；主系统独占根级 config/prd/tasks/...。
红线
绝对不写回 skills/ 源码目录、不污染他人子树。
05 创建治理文件
产出
manifest.json + agents/interface.yaml + reports/intent-context.json（三大必建文件）。
06 创建测试文件
产出
evals/trigger-tests.yaml，至少 5 个 should-trigger + 3 个 should-not-trigger。
07 写 references 深度内容
产出
模板、示例、完整指南（模板 / 治理清单 / 设计模式目录 / 产出路径规则 / 质检流程）。
08 写 CHANGELOG
产出
CHANGELOG.md 记录版本与变更。
09 跑 yao-meta-skill 质检
动作
六维度评分（metadata / ownership / boundary / assets / maintenance / trust），逐个修复不及格维度，复检。
门槛
Governance ≥ 90、Token ≤ 1000、Intent ≥ 90 才交付。
10 登记到索引
产出
更新 SKILLS-INDEX.md 与 核心Skill工程目录映射.md。

关键信息

触发条件

  • 创建全新的 Skill（从零或从经验提炼）
  • 审查已有 Skill 是否符合项目规范
  • 新聊天窗口中需让 Agent 了解「本项目怎么做事」

排除边界

  • 修改已有 Skill 的某个文件 → 直接改，无需本 Skill
  • 通用 Skill 工程方法论学习 → 用 yao-meta-skill
  • 功能开发任务 → 用 ai-dev-workflow

产出物

  • 新建 skills/<name>/ 全套标准布局
  • .ai-workflow/<name>/ 产出隔离子树
  • 更新 核心Skill工程目录映射.md

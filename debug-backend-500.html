Debug Backend 500（Docker 微服务）

系统性排查 Docker 部署的 Java 微服务后端 500 错误：Docker 环境感知 → 路由验证 → 日志分析 → 调用链追踪（Controller→Service→Dao）→ 空判断代码审查 → 部署层排查。非 Java 项目用 debug-problem。

执行全流程（5 阶段 SOP）

00 Docker 环境感知（必做先做）
动作
docker ps / inspect / logs 确认容器状态、重启次数、错误日志。容器刚重启过则靠代码审查兜底。
红线
不能假设本地 IDE 直跑相同；必须先做环境感知。
01 定位 Controller + 验证路由链
动作
grep 定位 Controller，逐层确认：前端请求 → 代理 → 网关路由 → context-path → Controller 注解路径一致性。
02 Docker 日志分析
动作
搜索异常关键词并显示上下文，按时间范围查询 NPE / DataAccessException 等。
03 调用链追踪
动作
并行读取 Controller → Service → SubService → Dao → DaoMapper.xml + Request/Response Model，全链路理解。
04 代码审查（空判断一致性）
重点
高危空判断与使用对象不一致（if(null!=A){use(B);}）、DB 返回 null 未保护、除零、异常被吞。
04.5 Docker 部署层（仍 500 时必做）
判断
「无可用服务」→ 未注册；「Connection refused」→ 端口不匹配；「500 变 404」→ context-path 错配。
05 根因定位与修复
产出
诊断报告 开发过程/<服务名>-500问题-<日期>.md：根因 + 错误代码 + 修复代码 + 验证命令（Docker compose restart 验证）。

关键信息

触发条件

  • 后端接口返回 500
  • Docker 容器服务出错
  • 需追踪 Controller→Service→Dao 调用链
  • 网关「无可用服务 / Connection refused」
  • 500 修复后变 404（context-path 错配）

排除边界

  • 前端路由 404 → debug-frontend-404
  • 性能/超时问题
  • 数据库死锁/连接池耗尽（非代码层）
  • 跳过 Stage-0 环境感知不适用

产出物

  • 开发过程/<服务名>-500问题-<日期>.md
  • 根因 + 修复 diff + 验证命令三段齐全
